Adding a New Model

How to Add a New AI Model

Three files. Zero plumbing. The game UI, training page, and simulator pick it up automatically.

Download a Working Example

Two starter templates ship with the project — pick the one that matches the kind of agent you want to build. Both are minimal, already registered, and runnable end-to-end.

Basic Minimax

Classical · Search-based

Depth-limited alpha-beta with a hand-crafted linear evaluator (harmony count, material, ring distance, opponent threat). No neural net, no gradients — just search + a few hardcoded weights.

basic_minimax.py — agent class registry.py — registration

Basic CNN

Learned · Value network

Tiny two-layer convolutional value network over an 8-channel 19×19 board tensor. Greedy 1-ply lookahead picks the move leading to the highest-valued successor state. Trained by self-play value-regression.

cnn_basic.py — agent class cnn_basic_training.py — training script registry.py — registration

Why two templates?

The agent interface here (choose_action(game)) is deliberately loose — it doesn't care whether your decision comes from tree search, a neural net, a lookup table, or a coin flip. Shipping both a classical search agent and a learned value network makes that clear by example: each template demonstrates a different part of the plumbing (registry entries with/without training, config files, log parsers, weight persistence), and together they cover the two branches almost every new agent falls into. Copy the one that matches your idea; the other tells you what the other branch of the registry looks like.

Swap the evaluator, widen the feature vector, change the net architecture, or replace the training loop — every extension point is isolated to one of these files.

1 Create the Agent Class

Create a new file in ai/rl/ (or ai/classical/). Your agent class must implement one method:

# ai/rl/my_agent.py

class MyAgent:
    def __init__(self, player=1, load=False, **kwargs):
        # Initialize your model / load weights when load=True
        ...

    def choose_action(self, game, verbose=False):
        """Return one legal action: ('plant', flower, r, c) or ('arrange', fr, fc, tr, tc)."""
        legal = game.get_legal_actions()
        # ... your logic ...
        return best_action

Interface requirements

  • choose_action(game, verbose=False) — takes a PaiShoGame instance, returns an action tuple.
  • Constructor should accept load=True to restore pre-trained weights at play time.
  • If your model needs to know which player it is, accept player= in __init__. The system will set agent.player before each move if you set "needs_player": True in the registry.

Optional methods for training

  • save_model(path=None) — persist weights to disk.
  • load_model(path=None) — restore weights from disk.
Tip: Use game.clone() before simulating moves in search. Never mutate the original game object.

2 Create a Training Script (optional)

If your model is trainable, add a script at ai/training/my_agent_training.py. It should:

  • Accept hyperparameters as CLI arguments (e.g. --n, --lr, --eps).
  • Print progress to stdout so the training dashboard can parse it.
  • Save the trained model to ai/params/MyAgent/.

Stdout format for live progress

The training dashboard parses stdout lines to show real-time stats. You can reuse an existing log format or add a new parser. The built-in parsers recognize these formats:

# PPO / Set Transformer format (log_parser: "ppo")
Episode 100/5,000 | eps=0.28 | P1 wins in 234 steps | W1=55 W2=40 D=5 | PL=0.023 VL=0.156 | params=12,544 | 3.21s

# TD Learning format (log_parser: "td_learning")
Episode 100/5,000 | eps=0.28 | P1 wins in 234 steps | W1=55 W2=40 D=5 | 3.21s

# Monte Carlo format (log_parser: "monte_carlo")
Episode 100/1,000 | Eps: 0.95 | P1 wins in 312 steps | Q-Table: 45,231 (+128) | Took 1.23s

# NEAT format (log_parser: "neat")
Generation 25
Tip: The easiest path is to print in the PPO format and set "log_parser": "ppo" in the registry. If you need a custom format, add a regex + extractor function to ui/training_manager.py's _LOG_PARSERS dict.

3 Register in ai/registry.py

Add one dictionary entry to the AGENTS list. Here's a minimal example for a trainable agent:

{
    "key": "my_agent",                   # unique identifier
    "display_name": "My Agent",          # shown in UI
    "description": "Short description",  # shown on model card
    "kind": "class",
    "module": "ai.rl.my_agent",          # Python import path
    "class_name": "MyAgent",             # class to instantiate
    "play_kwargs": {"player": 1, "load": True},  # constructor args for play
    "needs_player": True,                # set agent.player each turn
    "play_params": [],                   # runtime-tunable params (game UI)
    "model_path": "ai/params/MyAgent/model.pt",   # where weights live
    "training_script": "ai/training/my_agent_training.py",
    "training_params": [                 # shown on training page
        {"key": "episodes", "label": "Episodes", "type": "number",
         "default": 5000, "step": 500, "min": 1,
         "tooltip": "Number of self-play episodes"},
        {"key": "lr", "label": "Learning Rate", "type": "number",
         "default": 0.001, "step": 0.0001, "min": 0.00001, "max": 0.1},
        {"key": "resume", "label": "Resume", "type": "checkbox", "default": false,
         "tooltip": "Continue from saved checkpoint"},
    ],
    "training_cli_map": {                # param key -> CLI flag
        "episodes": "--n",
        "lr": "--lr",
        "resume": "--resume",
    },
    "total_episodes_key": "episodes",    # drives the progress bar
    "log_parser": "ppo",                 # reuse PPO's stdout parser
    "config_file": null,                 # editable config file path, or null
}

Registry field reference

FieldDescription
keyUnique machine name, used in URLs and CLI specs.
display_nameHuman-readable name shown in both UIs.
descriptionOne-line description for model cards.
kind"class" for agents, "function" for standalone functions, "inline" for random.
modulePython import path (e.g. ai.rl.my_agent). Lazy-imported on first use.
class_nameClass to instantiate from the module.
play_kwargsDict passed to the constructor when loading for play.
needs_playerIf true, the system sets agent.player = current_player before each move.
play_paramsRuntime-tunable parameters shown in the game UI (e.g. time budget). Each entry has: key, label, default, min, max, step.
model_pathRelative path to saved weights. Used by the training dashboard to show file info. null if none.
training_scriptRelative path to the training script. null if not trainable (agent won't appear on training page).
training_paramsHyperparameter definitions for the training UI. Each entry: key, label, type ("number", "text", or "checkbox"), default, and optional step, min, max, tooltip.
training_cli_mapMaps training_params keys to CLI flags for the training script.
total_episodes_keyWhich training_params key represents the total count (drives the progress bar).
log_parserKey into the parser registry in training_manager.py. Reuse "ppo", "td_learning", "monte_carlo", "neat", or add your own.
config_filePath to an editable config file (shown in training UI). null for most agents.

That's It

Once those three pieces are in place:

  • Your agent appears in the game page player dropdowns automatically.
  • If trainable, it appears on the training page with its own model card and hyperparameter form.
  • The simulator (python simulator.py --p1 my_agent) can load it with no changes.

Testing your agent

# Quick check: run a local game
python simulator.py --mode local --p1 my_agent --p2 random --n 1

# Train from the UI
python ui/server.py
# Open http://localhost:5000/train and select your model

# Run the test suite
python tests/test.py
